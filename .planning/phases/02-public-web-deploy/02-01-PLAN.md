<objective>
Identificar dónde se sirve hoy `bubabeautystudio.com` y dejar un pipeline de deploy reproducible. Si el hosting actual es AWS de la misma cuenta, extender `aws-email/` (o crear `aws-web/`) para gestionarlo con Terraform. - `scripts/deploy-web.sh` ejecutable que publica el sitio en un comando.
- Si es AWS en la misma cuenta: directorio `aws-web/` con Terraform (bucket + CloudFront + ACM + Route53 alias + deploy.sh).
</objective>

<task type="auto">
  <name>Task 1: Diagnosticar el hosting actual de bubabeautystudio.com</name>
  <files>.planning/notes/web-hosting.md</files>
  <action>
Ejecutar batería de checks:

```bash
mkdir -p .planning/notes
cat > /tmp/host-check.sh <<'EOF'
set -e
echo "=== DNS ==="
dig bubabeautystudio.com +noall +answer
dig www.bubabeautystudio.com +noall +answer
echo
echo "=== HTTP headers (apex) ==="
curl -sI https://bubabeautystudio.com | head -25
echo
echo "=== HTTP headers (www) ==="
curl -sI https://www.bubabeautystudio.com | head -10
echo
echo "=== Current served content check ==="
curl -s https://bubabeautystudio.com | head -20
echo
echo "=== Route53 records ==="
ZONE=$(aws route53 list-hosted-zones --query 'HostedZones[?Name==`bubabeautystudio.com.`].Id | [0]' --output text)
aws route53 list-resource-record-sets --hosted-zone-id "$ZONE" --query 'ResourceRecordSets[?Type!=`NS` && Type!=`SOA`].[Name,Type,AliasTarget.DNSName,ResourceRecords[0].Value]' --output table
EOF
bash /tmp/host-check.sh > /tmp/host-report.txt 2>&1
cat /tmp/host-report.txt
```

Escribir en `.planning/notes/web-hosting.md` un resumen con:
1. Provider identificado (GitHub Pages / Netlify / Vercel / CloudFront / S3 website / Cloudflare / otro) basado en headers `server`, `via`, `x-served-by` y en los records Route53.
2. Apex record: A / ALIAS / CNAME y hacia dónde apunta.
3. Si hay algún CDN delante: cómo se invalida la cache.
4. Dónde viven los archivos fuente (repo que publica, si es obvio por headers `x-github-request-id` o similares).
5. Recomendación de deploy pipeline basada en el provider.

  <done>Provider identificado con evidencia (headers / records). Si no es concluyente, se marca "TBD — preguntar al usuario" explícitamente.</done>

<task type="checkpoint:decision" gate="blocking">
  <decision>¿Cómo desplegamos? Depende del hosting detectado en Task 1.</decision>
  <context>Opciones típicas según lo encontrado en el diagnóstico. El usuario elige el camino.</context>
  <options>
    <option id="reuse-github-pages">
      <name>Hosting actual es GitHub Pages → publicar desde rama gh-pages o /docs</name>
      <pros>Cero coste extra, cero infra nueva, ya hay un pipeline. Solo hay que asegurarse de que la rama/carpeta correcta recibe los archivos.</pros>
      <cons>No hay CDN propio → invalidación gestionada por GitHub (puede tardar minutos). Sin CSP/security headers custom.</cons>
    </option>
    <option id="reuse-external">
      <name>Hosting actual es Netlify/Vercel/Cloudflare Pages → subir vía su CLI</name>
      <pros>Ya está montado y funcional.</pros>
      <cons>Añade un proveedor externo al stack; hay que ver si está vivo y con cuenta accesible.</cons>
    </option>
    <option id="migrate-aws">
      <name>Migrar a AWS S3 + CloudFront en la misma cuenta (clon de `aws-email/admin.tf`)</name>
      <pros>Stack 100% AWS-native (cumple constraint), paleta CSP/HSTS consistente con admin, mismo proceso de deploy que el admin. Coste + ~0,50 USD/mes.</pros>
      <cons>Requiere migración DNS en Route53 (TTL bajo antes del switch). ~2h de trabajo adicional.</cons>
    </option>
  </options>
  <resume-signal>Indicar "Opción: reuse-github-pages" / "reuse-external" / "migrate-aws". Si no se sabe aún, pegar el contenido de web-hosting.md y lo decidimos juntos.</resume-signal>
</task>

<task type="auto">
  <name>Task 3: Crear scripts/deploy-web.sh según la decisión anterior</name>
  <files>scripts/deploy-web.sh</files>
  <action>
Según la decisión del checkpoint anterior, generar un script ejecutable:

**Para `reuse-github-pages`:**
Script que copia `index.html`, `privacy.html`, `js/*.js`, `img/`, `llms.txt`, `robots.txt`, `sitemap.xml` a una rama `gh-pages` (o `/docs` en main), commitea y pushea. Detectar cuál es el patrón actual.

**Para `reuse-external` (Netlify):**
`netlify deploy --prod --dir=.` tras `netlify login` + `netlify link`.

**Para `migrate-aws`:**
Crear primero `aws-web/` con Terraform (clon de `aws-email/admin.tf` adaptado para el apex del dominio + www alias), apply, luego `scripts/deploy-web.sh` = `aws s3 sync` + `aws cloudfront create-invalidation`. Este task se alarga — si se elige esta opción, dividirla en 02-01a (terraform aws-web) y 02-01b (deploy script) para no saltar el límite de 2-3 tasks/plan.

En todos los casos el script debe:
1. Ser idempotente (ejecutarlo dos veces no rompe nada).
2. Excluir archivos de desarrollo (`.planning/`, `.git/`, `aws-email/`, `admin/`, `ios-app/`, `scripts/`, `node_modules/`).
3. Terminar con un check HTTP (`curl -sI https://bubabeautystudio.com | head -3`) para verificar que el nuevo HTML está sirviéndose.
